# 教師後台規劃文件

更新日期：2026-03-19

## 目的

本文件用來整理教師後台接下來要解決的問題、資料模型方向、功能優先順序，以及實作原則。

本文件的定位不是最終設計稿，而是後續逐步完善的工作基準。

## 目前後台已具備的能力

- 教師登入
- 查看學生進度資料
- 依週次、活動類型、完成狀態、關鍵字篩選
- 單筆重設學生進度
- 控制首頁週卡片顯示 / 隱藏

## 目前後台仍不足的地方

### 1. 學生識別不足

目前主要依靠 `email` 識別學生。

這對系統可用，但對老師不夠直覺。老師真正需要看到的是：

- 班級
- 座號
- 姓名
- 可選的小字補充 email

### 2. 活動只有資料紀錄，沒有活動主檔

目前系統主要依 `activity_key` 判斷活動，例如：

- `typing_task_2`
- `typing_task_3`
- `quiz_task_6`

這可用，但後台若要做：

- 顯示活動中文名稱
- 決定是否列入成績
- 設定活動權重

就需要正式的活動主檔，而不是只靠字串規則推測。

### 3. 成績評比規則尚未建模

目前只有：

- 闖關完成狀態
- 關卡進度
- quiz 分數

但還沒有：

- 哪些活動算平時成績
- 哪些活動算測驗成績
- 各活動權重
- 最後如何產生期末成績預覽

### 4. 資料量增加後缺少分頁與匯出

現在筆數少，整頁載入可接受。

之後若有：

- 多年級
- 多週次
- 多活動
- 全學期資料

就需要：

- 分頁
- 排序
- 匯出 CSV

## 建議資料模型

### 一、學生主檔 `student_profiles`

建議新增：

- `user_id`
- `email`
- `class_code`
- `seat_no`
- `display_name`
- `student_code`
- `role`
- `updated_at`

#### 顯示規則

後台表格優先顯示：

- `student_code + display_name`

例如：

- `30108 王小明`

email 則作為輔助資訊。

#### 實作策略

第一版建議由老師維護名單，不要讓學生自行填寫，避免資料品質不穩。

### 二、活動主檔 `activity_catalog`

建議新增：

- `activity_key`
- `grade`
- `week_code`
- `activity_type`
- `title`
- `max_score`
- `weight`
- `is_counted`
- `sort_order`
- `updated_at`

#### 用途

- 將 `activity_key` 對應成中文活動名稱
- 定義活動類型
- 控制是否列入成績
- 決定成績計算權重

### 三、現有活動紀錄 `student_progress`

目前繼續保留，定位為：

- 某學生
- 某週
- 某活動
- 的最新狀態

#### 現有欄位用途

- `current_level`：typing 類活動進度
- `completed`：是否完成該活動
- `score`：quiz 類活動分數
- `updated_at`：最後活動時間

## 成績評比方向

目前先不急著把成績規則寫死到 SQL。

建議先規劃成可調整模型。

### 建議的成績組成

- 平時任務完成率
- 測驗平均分數
- 課堂參與 / 準時完成情形

### 第一版可落地做法

- 後台先提供「成績預覽」
- 權重先寫在設定層，不直接寫死於資料表
- 等老師實際使用後再決定是否需要成績快照

## 功能優先順序

### 第 1 階段：先解決識別問題

1. 建立 `student_profiles`
2. 後台顯示班級座號姓名
3. 支援依班級、姓名、座號搜尋

### 第 2 階段：活動正式化

1. 建立 `activity_catalog`
2. 後台顯示活動中文名稱
3. 明確區分 typing / quiz / project

### 第 3 階段：後台查詢能力提升

1. 分頁
2. 排序
3. 總筆數顯示
4. CSV 匯出

### 第 4 階段：成績功能

1. 成績規則
2. 成績預覽
3. 匯出成績
4. 必要時加入成績快照

## 其他值得提早納入規劃的功能

### 1. 教師角色管理

目前 `is_teacher()` 是 email 白名單。

短期可用，長期建議改成角色表或 profile role 欄位。

### 2. 批次操作

後台遲早會需要：

- 批次重設
- 批次切換週卡片顯示
- 批次匯出

### 3. 操作紀錄

只要後台開始有：

- 重設資料
- 顯示 / 隱藏卡片
- 批次操作

就應考慮保留操作紀錄。

## 目前建議的下一步

若要繼續完善後台，建議優先做：

1. `student_profiles` 規格與 SQL
2. 後台表格學生欄位改版
3. `activity_catalog` 規格與 SQL

先把「人」與「活動」正式建模，再談成績與報表，結構會穩很多。
