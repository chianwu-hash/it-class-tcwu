# 教學網站開發與維護指引

更新日期：2026-03-19

## 目的

本文件用來規範本專案後續開發方式，降低結構再度失控的風險。

目標是做到：

- 不論由哪位開發者接手，都能快速理解架構
- 不論由哪個 AI 協助，都能依同一套原則實作
- 新功能可以逐步擴充，而不是再回到複製貼上式開發

## 核心原則

### 1. 小步前進

每次只改一小塊。

避免一次同時做：

- 新功能
- 重構
- 文案修改
- 視覺調整

同一輪若混太多目標，除錯會失真。

### 2. 先寫規格，再改程式

任何中型以上變更，都應先寫文件，至少說清楚：

- 要改什麼
- 不改什麼
- 驗收標準
- 失敗時怎麼回退

建議文件位置：

- `docs/refactor/`
- `docs/`

### 3. 先訂測試規格，再實作

每次重構前，先定義：

- 要測哪些頁面
- 哪些行為必須維持不變
- 哪些算成功
- 哪些一出現就必須中止

若沒有測試規格，不應直接大改。

### 4. 優先抽共用模組，不再複製貼上

目前已經建立的共用層：

- `shared/auth.js`
- `shared/week-visibility.js`
- `shared/typing-challenge.js`
- `shared/course-navbar.js`
- `shared/quiz-module.js`

新功能若和這些責任相近，優先考慮擴充共用模組，而不是在頁面內重新寫一套。

### 5. 保持網址與教學內容穩定

重構原則是：

- 不改網址，除非另外決議
- 不改學生既有操作流程，除非另外決議
- 不因重構而改變教學內容

## 目前架構分工

### 一、頁面層

頁面檔負責：

- HTML 結構
- 頁面專屬內容
- 最小掛接程式

頁面檔不應再自行持有大量共用邏輯。

### 二、共用模組層

共用模組負責：

- auth
- navbar
- 首頁週卡顯示
- typing challenge
- quiz 核心

若某功能已在兩頁以上重複，應優先考慮抽到共用模組。

### 三、專用 adapter 層

當某頁有特殊需求，但核心流程仍可共用時，應採 adapter 模式。

例如：

- `grade3/week06.quiz-adapter.js`

這類檔案的責任是：

- 接住該頁的特殊 selector
- 提供該頁專屬文案
- 掛接共用模組

而不是再自己重寫核心流程。

## 對 AI 協作的具體要求

### 1. 先讀文件，再動手

接手前至少先讀：

- `docs/refactor/site-refactor-plan.md`
- `docs/refactor/test-spec.md`
- 與該功能相關的設計文件

### 2. 不可直接大範圍重寫高風險 HTML

高風險頁面例如：

- `grade3/week06.html`
- `admin-progress.html`

若需調整，應優先：

- 抽外部 JS
- 新增 adapter
- 只做最小掛接

不要直接在內嵌大段中文 script 上做大範圍重組。

### 3. 任何重構都要保留回退能力

建議做法：

- 每個階段一個 commit
- 每個階段有明確驗證頁面
- 驗證通過再進下一刀

### 4. 不把未驗證的新邏輯直接推到正式站

流程應為：

1. 本機測試
2. 依測試規格驗證
3. commit
4. 視風險決定是否 push

### 5. 先隔離風險，再擴充功能

若某頁面已知高風險，不要在同一輪同時：

- 抽模組
- 改文案
- 換資料
- 改版型

先把責任切開。

## 編碼與檔案處理規則

### 1. 全專案統一使用 UTF-8

文字檔必須使用 UTF-8。

### 2. 腳本讀寫檔案時明確指定 UTF-8

例如 Python：

- `encoding=\"utf-8\"`

### 3. 不要只靠終端亂碼判斷檔案壞掉

Windows / PowerShell 在中文環境下，可能出現：

- 終端顯示亂碼
- 瀏覽器實際正常

判斷是否真的壞檔，要以：

- 瀏覽器實際畫面
- 語法檢查
- git diff

綜合判斷。

### 4. 高風險頁面避免 shell 批次改大段中文 script

若頁面內含：

- 大量中文文案
- 大量內嵌 JS
- 多段互動流程

應優先：

- 抽出外部模組
- 用最小 patch 掛接

## 建議開發流程

### Step 1：先寫問題與目標

至少寫清楚：

- 問題是什麼
- 這次只要解決什麼
- 哪些不處理

### Step 2：先盤點影響檔案

不要邊改邊找。

先列出：

- 會改哪些頁面
- 會改哪些 shared module
- 會不會影響 Supabase / Vercel / Netlify

### Step 3：先定測試規格

至少列：

- 測哪幾頁
- 哪些功能不能退化
- 驗收條件

### Step 4：實作

每次實作應維持：

- 小 patch
- 小 commit
- 可回退

### Step 5：本機驗證

優先在：

- `http://localhost:3000`

做驗證，不用 `file://`。

### Step 6：再決定是否 push

若當天有正式教學需求，優先保守處理，不要白天上線高風險改動。

## 目前已知高風險區

### 1. `admin-progress.html`

已知曾出現：

- 切分頁回來按鈕失效

目前解法是：

- 切回頁面時自動刷新

這是可接受解，但後續仍可再優化。

### 2. `grade3/week06.html`

這頁同時包含：

- auth
- quiz
- Email 任務
- typing challenge

後續若再調整，應優先透過 adapter 與外部模組進行，不要直接重組整頁內嵌 script。

## 後續建議

接下來若要繼續演進，建議順序：

1. 完善教師後台資料模型
2. 導入 `student_profiles`
3. 導入 `activity_catalog`
4. 擴充 quiz module 的共用能力
5. 將 typing 細部錯誤提示升級成共享標配

## 結語

本專案接下來的目標，不是追求一次重做，而是：

- 每次只整理一小塊
- 每塊都先定測試規格
- 驗證後再推進

只要持續遵守這個節奏，不論是人或 AI 接手，都能在不打亂既有功能的前提下穩定維護。
